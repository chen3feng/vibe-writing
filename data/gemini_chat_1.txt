## Q1: 浏览器中运行dos游戏用的是什么技术

在浏览器中运行 DOS 游戏，并不是让浏览器“直接”识别 DOS 程序，而是通过
在浏览器内部构建一个模拟的计算机环境
来实现的。
其核心技术链条可以概括为：
DOSBox (模拟器内核) + Emscripten (编译器) + WebAssembly (执行引擎)
。
核心技术架构
1. 模拟器内核：DOSBox
DOSBox 是目前最成熟的开源 DOS 模拟器，能够模拟 Intel x86 CPU、显卡（如 VGA/VESA）和声卡（如 Sound Blaster）。它是浏览器运行 DOS 游戏的基础“灵魂”。
2. 编译技术：Emscripten
由于 DOSBox 是用 C++ 编写的，而浏览器只能运行 JavaScript 或 WebAssembly。
Emscripten
是一个神奇的编译器，它能将 C/C++ 代码编译成能在网页上运行的代码。
它不仅转换代码，还提供了一套“虚拟系统接口”，让 DOSBox 觉得自己在读写真实的硬盘、接收真实的键盘输入。
3. 执行引擎：WebAssembly (Wasm)
这是目前浏览器运行复杂应用的关键。
高效性
：它是一种底层的二进制指令格式，运行速度接近原生程序。
流程
：Emscripten 将 DOSBox 编译为
.wasm
文件。当你打开网页时，浏览器下载这个文件并以极高的性能运行，从而保证游戏画面不卡顿、声音不延迟。
运行流程图解
加载阶段
：浏览器通过 HTTP 下载编译好的
DOSBox Wasm 模块
和
游戏镜像文件
（通常是
.zip
或
.img
）。
虚拟化阶段
：Wasm 模块在浏览器沙盒中启动，划出一块内存作为虚拟机的“RAM”，并建立一个
虚拟文件系统
（IndexedDB）来保存你的存档。
渲染阶段
：
画面
：使用
Canvas API
或
WebGL
将 DOS 的像素画面实时绘制在网页上。
声音
：使用
Web Audio API
模拟老式声卡的合成音。
交互
：通过 JavaScript 监听你的键盘和鼠标事件，并传递给虚拟的 DOS 环境。
常见的现成方案
如果你想在自己的网页上部署，通常会用到以下这些集成好的库：
技术/项目
说明
JS-DOS
目前最流行的封装库，它把复杂的 Emscripten 配置简化成了几行 JavaScript 代码。
Em-DOSBox
DOSBox 官方代码的 Emscripten 移植版本，很多在线游戏网站（如中文 DOS 游戏馆）的底层。
DOS.Zone
一个更高级的平台，支持 Windows 95 模拟甚至 3D 加速（3dfx 模拟）。
Export to Sheets
一句话总结：
这本质上是
用 C++ 写了个模拟器，再把它强行翻译成浏览器听得懂的二进制指令（WebAssembly）
，让你的 Chrome 瞬间变身成一台 1990 年代的 486 电脑。

## Q2: Jslinux呢

JSLinux 与运行 DOS 游戏的技术路径非常相似，但它的“野心”更大。如果说 DOSBox 是在浏览器里模拟一台 486 电脑，那么
JSLinux 则是直接用 JavaScript 模拟了一整套现代计算机硬件。
它是传奇程序员
Fabrice Bellard
（也是 QEMU 和 FFmpeg 的作者）的作品。以下是它的核心技术逻辑：
1. 硬件级仿真 (PC Emulator in JS)
与 DOSBox 这种专注于游戏指令集的模拟器不同，JSLinux 是一个
全功能虚拟机
。
CPU 模拟
：它用 JavaScript 实现了一个完整的 x86、RISC-V 或 Arm64 CPU 指令集。这意味着它可以运行任何编译给这些架构的操作系统。
外设模拟
：它模拟了可编程中断控制器（PIC）、定时器（PIT）、串口（UART）以及 PCI 总线。
2. 内存与文件系统管理
Typed Arrays
：JSLinux 利用 JavaScript 的
ArrayBuffer
和
Uint32Array
来开辟一块连续的内存空间，作为虚拟机的“物理内存”。
块设备模拟
：Linux 需要硬盘。JSLinux 通过将大文件切片加载，或者利用浏览器的
IndexedDB
模拟出一个持久化的块设备（硬盘），让 Linux 能够执行
mount
等挂载操作。
3. 操作系统镜像 (OS Images)
JSLinux 不仅仅是模拟器，它还包含了高度定制的操作系统镜像：
核心组件
：通常包含 Linux Kernel、BusyBox（提供基础 Shell 命令）以及 C 语言编译器（TinyCC）。
启动过程
：浏览器加载 JS 脚本 → 初始化虚拟硬件 → 将 Kernel 镜像载入虚拟内存 → CPU 仿真器从跳转地址开始执行代码。
JSLinux vs DOS 游戏模拟 (JS-DOS)
特性
DOS 游戏模拟 (DOSBox)
JSLinux
侧重点
侧重于显卡、声卡和游戏兼容性
侧重于 CPU 指令和系统内核稳定性
核心技术
Emscripten
(将 C++ 转换成 Wasm)
纯手写 JavaScript/Wasm
(直接模拟 CPU)
运行内容
DOS 程序 (.exe / .com)
完整的 Linux 内核、Windows 2000 等
性能消耗
中等，现代浏览器压力不大
较高，尤其是模拟现代多核 CPU 时
Export to Sheets
为什么 JSLinux 很有名？
在 JSLinux 诞生之初（2011年），大多数人认为 JavaScript 只能做表单验证或简单的动画。Bellard 通过 JSLinux 证明了
JavaScript 的执行效率已经足以支撑起一个完整的操作系统内核
。
现在的 JSLinux 甚至支持了
VirtIO
驱动，可以通过浏览器的 WebSocket 实现虚拟机的联网，这在技术层面上几乎把浏览器变成了一个“云桌面”终端。

## Q3: 我看到用jslinux技术也能运行Windows xp和Windows 2000

没错，你的观察非常敏锐。能在浏览器里看到 Windows 2000 甚至 XP 的滚动条，确实让人有种“时空错乱”的技术震撼感。
JSLinux 之所以能跑 Windows，本质上是因为它已经从一个简单的脚本演变成了一个
全功能虚拟化平台
。
为什么 JSLinux 能运行 Windows？
运行 DOS 游戏只需要模拟 CPU 的一部分指令和特定的多媒体硬件，但运行 Windows XP 这种现代操作系统，需要满足以下几个硬性指标：
1. 完整的 x86 架构模拟
Windows XP 依赖于复杂的 CPU 特性，如
保护模式 (Protected Mode)
、
虚拟内存管理 (Paging)
和
特权级切换 (Ring 0/3)
。
JSLinux 实现了一套近乎完整的 x86 指令集仿真器。
它能够处理 Windows 内核需要的内存分页表（Page Tables）和中断处理。
2. 核心硬件的“欺骗”
Windows 启动时会扫描硬件。JSLinux 为其伪造了一套标准的 PC 工业组件：
IDE 控制器
：让 Windows 以为自己在从硬盘读取数据。
VGA 显卡
：提供标准的图形输出。
RTC (实时时钟)
：保证系统时间正常。
以太网卡 (通常是 NE2000 兼容)
：通过浏览器的 WebSocket 代理，甚至能让虚拟的 XP 上网。
3. 性能瓶颈的突破：Wasm
早期 JSLinux 是纯 JavaScript 编写的，跑 Windows 2000 极慢（启动可能要几分钟）。
现在的版本大量使用了
WebAssembly (Wasm)
。
Wasm 让 CPU 的指令模拟（即“解释” x86 指令的过程）速度提升了数倍。
这使得 Windows XP 的启动时间缩短到了可接受的范围内。
技术挑战：为什么 XP 是个“天花板”？
虽然能跑 XP，但你会发现运行过程有些吃力，主要原因有两点：
缺少 GPU 加速
：
JSLinux 模拟的是基础 VGA 显卡，没有 3D 加速。这意味着你在里面玩《扫雷》很顺畅，但想跑 Direct3D 游戏（如《反恐精英》）几乎是不可能的。
内存与指令效率
：
x86 指令非常复杂（CISC）。在浏览器里用软件去逐条解释这些指令，天然存在转换损耗。运行 XP 这种已经具备初步多任务处理能力的系统，对主机的 CPU 压力非常大。
除了 JSLinux，还有哪些同类方案？
如果你对这种“浏览器里的操作系统”感兴趣，还可以关注以下技术：
v86
：这是一个非常流行的开源项目，它不仅能跑 Linux 和 Windows 98/2000，甚至支持保存“虚拟机快照”。你可以直接在浏览器里从别人保存好的状态瞬间唤醒 XP。
Copy.sh
：一个基于 v86 的在线集成环境，预装了从 FreeDOS 到 Windows 1.01 再到 Arch Linux 的各种镜像。
总结
JSLinux 运行 Windows 证明了：
只要模拟器够完整，底层执行环境（浏览器）够快，软件可以完全脱离物理硬件的限制。
你是在尝试在自己的网页里嵌入这些系统吗？还是单纯对这种虚拟化方案感兴趣？

## Q4: Copy.sh是什么技术

Copy.sh
其实并不是一种全新的底层技术，而是一个极具代表性的
开源集成项目
。它的核心引擎被称为
v86
。
如果你把 JSLinux 看作是一个天才程序员的“个人杰作”，那么 Copy.sh (v86) 就是目前 Web 虚拟化领域最流行的
通用工具箱
。
1. 核心引擎：v86
Copy.sh 的本质是一个基于 JavaScript 的
x86 PC 模拟器
。它与 JSLinux 的思路一致，但在实现细节上有所不同：
指令翻译 (JIT-less Emulation)
：v86 将 x86 指令解释为 JavaScript 或 WebAssembly 执行。它模拟了完整的 Intel 80386 指令集。
硬件全面仿真
：它不仅模拟 CPU，还模拟了软盘驱动器、IDE 硬盘、键盘、鼠标（PS/2）、甚至是
Sound Blaster 16 声卡
和
NE2000 网卡
。
状态快照 (Snapshots)
：这是 Copy.sh 最强大的地方。它可以将当前虚拟机的内存状态完整保存为一个文件。你打开网页时，它不是从头“开机”，而是直接恢复快照，实现
秒开 Windows 98
。
2. 关键技术栈
Copy.sh 之所以流畅，主要归功于以下三剑客：
WebAssembly (Wasm)
：
由于纯 JavaScript 处理二进制指令非常吃力，v86 的核心执行循环是用 Rust 或 C 编写并编译为
Wasm
的。这使得它的运行速度比早期的 JS 模拟器快了 5-10 倍。
Web Workers
：
为了防止模拟器运行导致网页“假死”，Copy.sh 将复杂的 CPU 计算放在后台的
Web Worker
中运行，主线程只负责渲染画面和接收输入。
SharedArrayBuffer
：
用于在不同线程间高速共享内存数据，保证模拟的硬件设备（如显卡）能实时读写“物理内存”。
3. Copy.sh 与 JSLinux 的区别
虽然它们都能在浏览器里跑系统，但定位不同：
特性
JSLinux
Copy.sh (v86)
开发者
Fabrice Bellard (大神孤品)
开源社区 (基于 Rust/JS)
兼容性
极高，甚至支持 RISC-V 和高级内核
侧重于 x86 传统硬件，兼容性极广
可定制性
闭源内核，难以二次开发
高度模块化
，你可以轻易地把它集成到自己的网站中
特色功能
极致的精简和算法优化
强大的
快照功能
和丰富的图形化配置
Export to Sheets
4. 为什么大家喜欢用 Copy.sh？
Copy.sh 的流行主要是因为它把“黑科技”变成了“易用品”：
即插即用
：它提供了一个现成的在线环境，你只需要上传一个 ISO 镜像，它就能在网页里帮你启动系统。
网络穿透
：它通过 WebSocket 代理（如
wsproxy
），让浏览器里的老系统真的能通过你现在的网络去访问真实的互联网。
Rust 驱动
：新版的 v86 使用 Rust 编写，利用了 Rust 的内存安全和高性能，使得模拟更加稳定。
一句话总结：
Copy.sh 是一个利用
WebAssembly
性能红利，将
Rust 编写的 x86 模拟器
搬进浏览器的开源项目。它证明了浏览器已经不再只是一个文档浏览器，而是一个
通用的硬件执行平台
。

## Q5: 有哪些早期的游戏机模拟器，比如nes的

早期的游戏机模拟器（如 NES/FC、Game Boy、SNES）进入浏览器，比 Windows XP 这种大型系统要早得多。
这些模拟器的技术路线经历过从
JavaScript 暴力破解
到
WebAssembly 精准仿真
的演变。
以下是几个具有代表性的技术方案：
1. JSNES：网页 NES 模拟器的开山鼻祖
这是最著名的开源 JavaScript NES 模拟器。
技术原理
：纯 JavaScript 编写。它手动模拟了 6502 处理器、PPU（图形处理单元）和 APU（音频单元）。
特点
：在 WebAssembly 普及之前，JSNES 是行业标准。虽然纯 JS 的效率不如原生，但对于 NES 这种只有
1.79
MHz
的 8 位机来说，现代浏览器的 JS 引擎通过 JIT（即时编译）足以流畅运行。
2. Emscripten + Libretro (RetroArch)
这是目前最强大的“全能型”方案，也是很多在线游戏站点的底层架构。
技术原理
：利用
Emscripten
将 C/C++ 编写的成熟模拟器内核（如
Nestopia
跑 NES，
Snes9x
跑超任）直接编译为
WebAssembly
。
代表作：RetroArch Web Player
。它不是针对某一个机型，而是一个框架，通过加载不同的“Core”（核心），可以在浏览器里玩转从 Atari 到 PS1 的几乎所有早期主机。
3. GameBoy-Online
背景
：由 Grant Galitz 开发，是最早证明网页能流畅跑掌机游戏的案例之一。
技术细节
：它利用了早期的
Web Audio API
来重现 Game Boy 特有的 8-bit 电子音。

为什么这些模拟器比 JSLinux 跑得顺？
架构简单
：NES 的 6502 处理器指令集非常精简，只有几十条指令；而 x86（JSLinux）有数千条指令且极其复杂。

：NES 只有
2

渲染直观
：早期游戏的画面是像素阵列，可以直接映射到 HTML5 的
<canvas>
标签上，通过
putImageData
函数就能以
60 FPS
的帧率刷新。
趣味冷知识：AssemblyScript
现在很多现代的网页版模拟器开始使用
AssemblyScript
编写。这是一种语法和 TypeScript 几乎一样，但直接编译成
WebAssembly
的语言。它让开发者既能享受类似 JS 的开发体验，又能获得接近 C 语言的运行性能，是目前开发网页版模拟器的新宠。

## Q6: 非web版的呢

非 Web 版（原生版）模拟器的技术，本质上是从“翻译”向“虚拟化”进军的过程。它们直接运行在操作系统（Windows, macOS, Linux）之上，不需要经过浏览器和 WebAssembly 的层层转手，因此性能更强，技术手段也更硬核。
原生模拟器的核心技术主要分为以下三类：
1. 解释执行 (Interpretation)
这是最基础、兼容性最好的技术，早期的 NES 和 Game Boy 模拟器大多采用这种方式。
原理
：模拟器程序就像一个“翻译官”，逐条读取游戏 ROM 里的机器指令。比如读取到一条
LDA #$01
，模拟器会在 C++/Rust 代码里执行一个对应的函数，改变模拟出来的“虚拟寄存器”的值。
优缺点
：开发简单，逻辑清晰；但
效率极低
，因为 CPU 处理每一条指令都要经过大量的逻辑判断。
2. 动态重编译 (Dynamic Recompilation / JIT)
这是现代高性能模拟器（如
RPCS3
跑 PS3、
Dolphin
跑 GC/Wii、
PPSSPP
跑 PSP）的核心。
原理
：模拟器不再一条条翻译，而是
成块地翻译
。它会预测游戏接下来要运行的一段代码，直接将其转换成你电脑 CPU（比如 Intel 或 AMD 的 x86-64）能直接识别的指令并缓存起来。
效果
：第一次加载可能会有微小卡顿，但之后运行速度飞快，接近原生程序的效率。
3. 硬件加速渲染 (Hardware Abstraction Layers)
非 Web 版模拟器可以直接调用显卡的底层 API，而不必受限于浏览器的 WebGL：
图形 API
：使用
Vulkan
、
DirectX 12
或
OpenGL
。这些 API 允许模拟器直接操作显存，实现
内部分辨率翻倍
（让原本模糊的 PS2 游戏画质达到 4K）。
着色器模拟
：利用 GPU 的计算能力来模拟老主机特殊的发色数或图形特效。
4. 甚至还有“硬件级”模拟：FPGA
如果你追求 100% 的还原度，甚至不想要“软件模拟”，那就涉及到了
FPGA (现场可编程逻辑门阵列)
技术。
代表作
：
Analogue Pocket
、
MiSTer FPGA
。
技术本质
：它不是在跑软件，而是通过硬件编程，把 FPGA 芯片内部的电路物理性地重组为和原版主机（如 MD 或 SNES）一模一样的电路。
优势
：
零延迟
。因为它在物理层面就是那台机器，而不是在“演”那台机器。

总结
非 Web 版模拟器的本质是“压榨性能”。
由于没有浏览器沙盒的限制，它们可以使用多线程并行计算、直接访问内存寄存器、以及调用最新的 GPU 驱动。这也是为什么你的电脑可以流畅运行 PS3 模拟器，但浏览器目前最多只能勉强跑跑简单的 PS1 游戏。

## Q7: 任天堂的模拟器有哪些

任天堂的模拟器生态非常成熟，从 80 年代的红白机（NES）到现在的 Switch，每一个世代都有极具代表性的“神级”模拟器。
在 2026 年的今天，根据你所使用的平台和想要玩的游戏，可以分为以下几类：
1. 全能型（新手首选）
如果你不想每一个机型都装一个软件，
RetroArch
是绝对的王者。
技术原理
：它是一个“外壳”，通过加载不同的“Core（内核）”来运行不同机型。
适用机型
：NES, SNES, GB, GBC, GBA, N64 等所有早期主机。
优点
：统一的界面、支持成就系统（RetroAchievements）、强大的滤镜（让液晶屏模拟出老电视的扫描线）。

## Q8: 如何获得合法的游戏ROM

获取“合法”的游戏 ROM 是一个技术与法律边界交织的话题。在 2026 年，法律共识依然倾向于：
只有当你拥有正版游戏副本，并亲自通过硬件手段提取出的 ROM，才被广泛认为是合法的个人备份。
以下是获得合法 ROM 的主要途径：
1. 自行从原始卡带/光盘中“提取”（Ripping）
这是法律风险最低、最硬核的方式。你需要一些专门的硬件工具来读取卡带数据：
卡带读取器 (Cartridge Dumpers)
：
Retrode 2
：最经典的工具，支持 SNES 和 Mega Drive 卡带，还可以通过扩展插槽读取 GB/GBA 和 N64 卡带。它把卡带模拟成一个 USB 闪存盘，你只需像复制文件一样把
.sfc
或
.bin
文件拷出来即可。
GB Operator
：专门用于 Game Boy 系列卡带的读取器，小巧且带有精美的图形界面。
破解版主机
：
如果你有破解过的
3DS
或
Wii
，可以使用内置工具（如 3DS 的
GodMode9
）直接将插入的卡带或数字版游戏提取为
.3ds
或
.cia
文件，再转为模拟器可读的格式。
光盘镜像
：
PS1、PS2、世嘉土星等光盘游戏最简单，只需电脑带一个普通的 DVD 光驱，使用
ImgBurn
等软件将其抓取为
.iso
或
.bin/cue
格式即可。
2. 从官方合集或数字平台“提取”
很多游戏厂商会在 Steam、GOG 或主机平台上发布“复古合集”，这些合集内部其实往往就藏着原始的 ROM 文件。
SEGA Mega Drive Classics (Steam)
：这是目前最“良心”的例子。你购买这个合集后，进入游戏的安装目录（
steamapps/common/Sega Classics/uncompressed ROMs
），你会发现所有游戏的原始 ROM 就在文件夹里，你可以直接把它们拿出来放到任何模拟器里运行。
卡普空街机馆 (Capcom Arcade Stadium)
：通过一些 Python 脚本工具（如
Game Extraction Toolbox
），玩家可以合法地从 Steam 下载的文件中解析并提取出原始的街机 ROM。
SNK / NeoGeo 合集
：GOG 平台上销售的很多 SNK 经典游戏（如《合金弹头》）本质上就是官方打包的模拟器+ROM，文件结构通常很清晰。
3. 购买授权的“复刻”硬件
Analogue 系列 / MiSTer FPGA
：这些硬件通常支持直接插入原装卡带运行。虽然它们不是直接给你一个文件，但它们提供了一种无需违法下载就能在现代屏幕上获得完美模拟体验的途径。
4. 寻找“自制游戏” (Homebrew)
很多现代开发者在为老主机开发新游戏（Homebrew）。
在
Itch.io
或
PDROMs
等平台上，开发者会直接出售或免费提供
.nes
、
.gba
格式的游戏文件。这些 ROM 没有任何版权争议，是 100% 合法的。
⚠️ 特别提醒
“下载即违法”原则
：在大多数地区，即便你拥有该游戏的实体卡带，从网站上下载别人上传的 ROM 仍被视为侵权（因为你获取的不是“你自己的”备份）。
BIOS 文件
：像 PS1、PS2、Switch 等模拟器通常需要系统的 BIOS 固件才能运行。这些固件是版权保护极严的代码，
合法获取的方式只有一种
：从你自己拥有的实体主机里通过技术手段导出来。
总结建议
：
最稳妥且能避开法律麻烦的方案是：
购买 Steam 上的官方 Sega/Atari/Capcom 合集
，然后用简单的文件操作把里面的 ROM 提出来。这既支持了厂商，也让你拿到了可以跨平台运行的合法文件。
