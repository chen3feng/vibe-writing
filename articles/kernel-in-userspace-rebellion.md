# 把内核搬进应用：从自绘 UI 到 QUIC 的一次"用户态反叛"

> 每一个看似无关的技术潮流，拉远看，往往是同一种架构思想在不同层次上的重复投影。

如果你同时做过两件事——给桌面应用换过 UI 框架，和给服务端换过网络协议——你可能会在某个深夜突然打了个激灵：

Flutter 对 UI 做的事，和 QUIC 对 TCP 做的事，好像是一回事。

它们都在说同一句话：**"系统给的那套抽象不够用了，我自己来。"**

这不是比喻，而是一次结构上的同构。当你把两件事放在一张图上对照，它们的动机、路径、代价、甚至最终的归宿都惊人地一致。这篇文章就是要把这个同构讲清楚——因为一旦你看见它，你会在很多别的地方也看见它。它不只关于 UI，也不只关于网络。它关于软件世界里一条反复出现的规律：**曾经由操作系统提供的子系统，正在一层一层被重新实现成应用层的库。**

---

## 一、两场反叛，一张剧本

先从两件表面上毫无关系的事情说起。

### 1.1 UI 的反叛：从"借控件"到"借画布"

Windows 早期，一个按钮就是一个 `HWND`。点击、焦点、重绘、右键菜单、键盘导航、辅助工具识别——这些事情全部由 `user32.dll` 替你打理。MFC、WinForms、VB6 的开发者，真正写的只是业务逻辑，UI 的"物理"完全由系统负责。

但这套体系有一个致命弱点：**它是 Windows 专属的**。你在 macOS 上找不到 `HWND`，Android 上没有 `WndProc`。每个操作系统都有自己的一整套控件模型，风格不同、能力不同、bug 不同，甚至同一个控件在 Windows 不同版本里行为都不同。

于是到了 2010 年之后，一股"自绘"浪潮涌起：Qt Quick、WPF、Flutter、Jetpack Compose，它们不再使用系统控件。它们向系统只要**一个窗口、一块画布**，然后在这块画布内部，用 Skia 或类似引擎，自己画按钮、自己处理点击、自己管理焦点、自己实现文本输入——**UI 控件体系被整个从内核/OS 里搬出来，变成一个应用内的库**。

### 1.2 网络的反叛：从"借协议栈"到"借 UDP 口"

几乎同一时期，网络领域上演了一模一样的剧本。

过去几十年，应用写网络代码只需要一件事：打开一个 TCP socket。连接建立、拥塞控制、重传、顺序保证、ACK 聚合——全部由内核协议栈替你搞定。你的代码里看到的是一根干净的字节流管道。

但 Google 在 2012 年前后发现：TCP 升级太慢了。想换个拥塞算法要进 Linux 内核，从提交到主流发行版普及要以年计；中间盒（防火墙、NAT、负载均衡器）还会各种篡改 TCP 报文；多路复用、0-RTT 这类现代需求在 TCP 的框架里根本塞不进去。

于是 QUIC 诞生了。它只向内核要**一个 UDP socket**，然后在这个 socket 之上，用户态实现了完整的传输层——握手、加密、多路复用、拥塞控制、流控、丢包恢复，**协议栈被整个从内核里搬出来，变成一个应用内的库**（msquic、quiche、lsquic、google-quiche）。

### 1.3 同一句话的两种说法

把两件事并排放一下：

| 维度 | UI 世界 | 网络世界 |
|------|---------|----------|
| **被抛弃的系统服务** | 系统控件体系（`user32`、AppKit、View） | 内核传输层（TCP 协议栈） |
| **向系统保留的最小接口** | 一个 HWND / CALayer / Surface | 一个 UDP socket |
| **搬进应用的是什么** | Skia + 控件树 + 输入/焦点/IME 逻辑 | 握手 + 加密 + 多路复用 + 拥塞控制 |
| **典型代表** | Flutter、Qt Quick、WPF、Compose | QUIC（HTTP/3）、DPDK、XDP |
| **主要推手** | Google（Flutter）、Microsoft（WinUI）、Qt | Google、Meta、Cloudflare |

两边几乎是一部电影拍了两遍。接下来要问的是：**为什么？**

## 二、动机：为什么要把内核搬进应用

这场反叛不是一时兴起，它有极其一致的动机。把两边开发者的抱怨逐条对齐，会发现用同一套词就能覆盖两侧。

### 2.1 跨平台的一致性

应用开发者最痛的一件事，是"同一份代码，两套行为"。

- **UI 侧**：Windows 上的单选按钮是圆的，macOS 上默认看起来不一样，Android 上又有 Material 规范，iOS 上是 HIG 规范。想要"一次开发、到处一致"，系统控件永远做不到——因为它的目标本来就是"入乡随俗"。
- **网络侧**：Linux、Windows、macOS 的 TCP 实现各有细微差异；更糟的是网络路径上的中间盒会篡改 TCP 选项、拒绝陌生标志位、魔改 MSS。QUIC 把自己塞进加密的 UDP 里，中间盒只能看到一堆密文 UDP 报文，**无从篡改，也就无从差异化**。

一致性是用户体验和工程成本的底层需求。系统抽象为了"通用"做出的折中，对想要"可控"的专业应用来说就是噪音。

### 2.2 迭代速度

这是两边最相通的痛。

- 系统控件跟着 OS 大版本发布，修一个 bug 或加一个新特性，至少要等到用户升级到新系统——这是以**年**为单位的节奏。Flutter 自己发版，修 bug 推送 OTA，是以**周**甚至**天**为单位。
- TCP 的拥塞算法升级曾经是最漫长的工程之一：BIC、CUBIC、BBR 每一次上线都要跨越整个操作系统发行链。QUIC 把拥塞控制搬到用户态后，Google 在 Chrome 里换一次 BBRv3 的实验流量，只需要一次客户端更新。

当"控制权在谁手里"决定"你多久能改一个 bug"，选择自然是明显的。

### 2.3 精细控制

系统抽象是通用抽象，通用就意味着**不够深**。

- 想做 60 fps 的共享元素动画、iOS 那种橡皮筋回弹、毛玻璃模糊叠加交互反馈——系统控件不给这种口子。你只能"自己画"。
- 想做应用层感知的拥塞反馈、按流的优先级调度、在同一条连接上并行多个独立流、0-RTT 恢复会话——TCP 不给这种口子。你只能"自己实现传输层"。

换句话说，**系统只对 80% 的用户负责**。剩下那 20% 想要的特性，要么等下一代 OS，要么自己干。

### 2.4 抽象税

这是性能工程师的视角。

- **UI 的抽象税**：一万个按钮等于一万个 HWND。每个 HWND 都带着消息循环、命中测试、焦点管理、GDI 资源。当 UI 从"几十个控件的表单"变成"几千个元素的可视化仪表盘"，系统控件体系就是压死性能的那根稻草。自绘引擎把这些摊成一棵普通对象树 + 一次性的 GPU 提交，开销直接塌一个数量级。
- **网络的抽象税**：内核态协议栈意味着每个数据包都要穿越 user/kernel 边界，意味着 syscall、上下文切换、缓冲区拷贝。10 Gbps 以上的场景里，内核协议栈常常就是瓶颈。DPDK 干脆把网卡映射到用户态轮询，QUIC 则起码把协议逻辑拉进了用户态。

两边的逻辑一模一样：**当调用系统服务的开销超过服务本身的价值，就是反叛开始的时候。**

### 2.5 动机的总结

四条动机可以压缩成一句话：

> **"通用系统抽象"是为了所有人都能用而设计的，"专业应用"想要的东西是为了自己能用到极致而设计的——两者的目标在某个规模以上必然分道扬镳。**

这不是新鲜现象，**TCP 自己**当年就是这么诞生的——它是对"裸 IP + 应用自己做可靠性"的一次系统化反叛，只不过它成功到被写进了内核，成了下一代的默认。

每一代反叛，都是下一代的体制。

## 三、代价：把世界带进家里

反叛不免费，而且代价同构得令人发笑。

### 3.1 生态与可观测性的割裂

这是最直接的痛。

- **UI 侧**：Flutter 早期在 Windows 上没有屏幕阅读器支持，因为系统不知道它画的那块东西是"按钮"；无障碍框架（UIA / AX）看到的是一整块图像。IME 输入法、右键菜单、系统文本选择、拖放——都要一一重做。Flutter 为此不得不另外维护一棵"影子语义树"（semantic tree），专门喂给系统的无障碍服务。
- **网络侧**：QUIC 刚出来那几年，企业防火墙不认，负载均衡器不认，抓包工具不认。Wireshark 看到的是一堆加密 UDP，什么也看不出来。排查生产问题需要专门导出 SSLKEYLOGFILE 再解密，或者让应用自己打 qlog。

两边的本质都是：**"原来的生态工具链依赖于系统抽象，你一旦绕过系统，生态就看不见你了。"**

它们的补救方案也惊人地同构：
- Flutter 做 `SemanticsNode` 喂给 UIA。
- QUIC 做 `qlog`/`qvis` 喂给运维。

都在往回"给系统/运维工具造一个影子视图"，因为他们知道：不给，工具链就不配合你。

### 3.2 体积膨胀与冗余

- 一个空的 Flutter 应用安装包 15–25 MB 起步，因为 Skia、Dart runtime、文字排版引擎都得塞进去。Electron 更夸张，每个应用自带一整个 Chromium，一百多兆打底。
- 每个跑 QUIC 的应用都得带一份完整的传输层实现。Chrome 带一份、curl 带一份、nginx 带一份——本来一份内核 TCP 可以给整个系统共享，现在每个进程各自复制一份。

**"拉库进家"的代价就是每家都多一份行李**。

### 3.3 冷启动与内存

- Electron / Flutter 的启动比原生慢一截，驻留内存也高——因为它们每次都要拉起自己的小世界。
- QUIC 连接的建立需要初始化用户态的密钥、拥塞状态、流表，在高并发短连接场景下，CPU 负担比内核 TCP 要高；网卡 offload 也要重做。

### 3.4 调试的降级

这一点最让老工程师难受。

- 你没法用 Spy++ 去看 Flutter 的控件树，你得用 Flutter DevTools。
- 你没法用 Wireshark 直接读 QUIC，你得导密钥才能解密。

工具链的知识要重学一遍。新范式带来的不仅是新能力，也是新的"不熟"。

### 3.5 代价的同一句话

所有代价可以凝成一句话：

> **为了控制权，自带整个世界。**

这也是为什么每一次"用户态反叛"成功之后，总会跟着一轮"重新系统化"的运动——这是第六节的伏笔，我们放到最后讲。

## 四、同一张地图上的两条谱系

如果把"从系统手里拿多少、自己做多少"作为一条坐标轴，你会发现 UI 和网络世界在这条轴上分布得几乎是镜像的。

```
系统包办 ←—————————————————————————————————→ 应用自理

UI 侧：
  WinForms  WinUI 3   Qt Quick / WPF   Flutter    游戏引擎
  (HWND)   (DComp自绘)  (Skia自绘)     (自带一切)  (直写swapchain)

网络侧：
  POSIX TCP  内核 TLS   QUIC (user)    DPDK       RDMA
  (全内核)   (部分下沉)  (UDP之上)      (旁路内核)  (网卡直通内存)
```

仔细看，会发现几个有趣的对应关系。

### 4.1 Flutter ≈ QUIC：温和的反叛

它们都还需要系统兜底一件事：
- Flutter 画完像素后，还是要把这块 surface 交给 DWM / CoreAnimation / SurfaceFlinger 去和别的窗口合成上屏——**系统还管合成**。
- QUIC 再自由，它的 UDP 报文还是要走内核的 IP 路由、网卡驱动、硬件中断——**系统还管 L3/L2**。

它们只在"抽象栈的中上层"反叛，把最底下的"硬件接入层"仍然交给系统。

### 4.2 游戏引擎 ≈ DPDK：激进的反叛

游戏引擎（Unreal、Unity）早就不满足于"系统合成"了，它们直接拿 swapchain，自己决定 vsync 和 present，系统合成器几乎不掺和。VR/AR 引擎甚至自己做 timewarp 直接写 display controller。

DPDK 则把整个网卡映射进用户态，轮询收发，内核协议栈完全旁路——连 UDP 这一层都不用系统了。

这两者都走到了更极端的"用户态反叛"，因为他们对延迟和抖动的要求到了每一纳秒都要抠的程度。

### 4.3 RDMA：极端反叛

RDMA 让网卡绕过本机 CPU 直接读写远端内存，甚至连**本地 CPU 都被旁路了**。对应到 UI 世界，就是 VR 头显直接向显示芯片推像素、跳过 OS 图形栈的场景。

### 4.4 一个规律

**激进程度越高，一致性与可控性越强，生态整合度越低，调试越难。**

这是一条贯穿整个坐标轴的规律。你选择反叛到哪一步，其实是在选"我要为哪些代价买单"。

## 五、Electron 与 VPN：同一个剧本的喜剧版

既然聊到"把整个世界带进家里"，一对近乎滑稽的同构是绕不过去的：**Electron** 和 **OpenVPN**。

### 5.1 同样的"富家子弟"问题

- **Electron**：每个应用自带一整份 Chromium，内存起步 150 MB。被开发者拥抱（因为用 Web 技术开发桌面应用太爽了），被用户吐槽（为什么我一个记事本要吃掉 1 GB 内存）。
- **OpenVPN**：完整的加密协议栈跑在用户态，每个包都要在 user ↔ kernel 之间反复穿越，性能差到只能跑办公流量，数据中心互联完全没法用。

### 5.2 同样的救赎之道

两者都面临同一种救赎路径：**把系统重新拉回来**。

- **Tauri 之于 Electron**：别再自带 Chromium 了，用系统原生 WebView（WebView2、WKWebView）。二进制从 100 MB 降到几 MB，内存占用降一个数量级。
- **WireGuard 之于 OpenVPN**：别再在用户态做协议栈了，整个实现塞进 Linux 内核 net namespace。代码量少到千行级，性能和延迟直逼内核 TCP。

你看，这是**同一条救赎路径**：**"让系统重新参与，以换取性能和体积。"**

### 5.3 自绘 vs 反叛后的回归

这给我们透露了一个信号：**"拉库进家"并不是终点，而是循环里的一个阶段**。下一阶段是"把库交还给系统"。这正是下一节要展开的——当反叛者成功，它们会变成下一代的体制。

## 六、反叛的循环：从"用户态反叛"到"重新被系统吸收"

这是最有意思的一层。

### 6.1 TCP 自己就是前一次反叛的胜利者

今天我们把 TCP 看作"内核给的东西"，但 TCP 诞生时并不是——它最初是作为应用层可靠性协议的抽象被提出的。它之所以最终进入内核，是因为它**成功到所有人都需要它，而每家自己实现一遍太浪费**。

TCP 是上一场反叛的胜利者。它反叛的对象是 UUCP 时代那种"自己在应用层做重传"的手工活，胜利之后被系统吸收，变成了默认。

### 6.2 QUIC 正在被系统吸收

QUIC 走的是同一条路：
- **Windows** 已经在 OS 里自带 msquic，并把它暴露给 HTTP/3。
- **Linux 内核** 正在讨论内核态 QUIC 实现。
- **网卡厂商**（Mellanox、Intel）正在做 QUIC 硬件 offload——就像当年做 TCP offload 一样。
- **curl、nginx、浏览器**越来越多地直接调用"系统 QUIC"而不是自带一份。

这是一场可预见的"重新系统化"。五年之后，QUIC 很可能会像 TCP 一样被视为"系统自带"的东西，而不再是"每个应用自带一份"。

### 6.3 UI 的"重新系统化"也已经开始

这边的信号同样强烈：
- **WinUI 3 / System XAML**：微软在用 DComp 让原生控件也具备 Flutter 那种自绘能力，把"自绘 UI"重新吸收成系统能力。
- **Jetpack Compose**：Google 在把 Compose 推成 Android 官方 UI 标准，相当于把自绘范式写进 OS 默认。
- **SwiftUI**：Apple 完全没给 Flutter 机会，直接官方下场做了一套声明式+自绘+系统集成的方案，并把它做成 iOS/macOS 的第一公民。

这都是"反叛者的范式胜利后，被操作系统吸收再变成新默认"的典型动作。

### 6.4 这个循环的通用形态

抽象成一条定律：

```
  第一阶段：系统抽象够用 →
  第二阶段：系统抽象跟不上需求 →
  第三阶段：应用开始在用户态重新实现系统抽象（库化）→
  第四阶段：库化方案成为事实标准 →
  第五阶段：系统重新吸收这套方案，变成下一代默认 →
  （回到第一阶段）
```

TCP 走完了全程。QUIC 走到了第四到第五阶段的过渡。Flutter 走在第三到第四阶段。WinUI / Compose / SwiftUI 已经在做第五阶段的工作。

每一层抽象，几乎都在独立地重演这个循环：文件系统 → FUSE → 重新入内核；进程调度 → 用户态协程 → 内核调度器优化；中断处理 → DPDK 轮询 → 网卡 offload；DNS 解析 → 应用内 DoH/DoT → 系统 DoH 服务。

**"用户态反叛"从来不是异常，它是软件系统演化的常态。**

## 七、为什么这个同构值得被看见

讲到这里，"UI 和网络长得一样"这件事本身已经有点意思了。但更重要的是，**一旦你学会识别这个模式，你会在更多地方看见它**。

### 7.1 它是一个判断新技术"合理性"的视角

当你下一次看到某个新技术在说"我们把 X 从内核/系统拉到了用户态"，你可以立即把它放进这张地图：

- 它在反叛什么系统抽象？
- 为什么现在反叛？是因为一致性、迭代速度、精细控制，还是性能？
- 它的体积/启动/生态代价是什么？
- 它会不会像 TCP/QUIC 那样最终被系统重新吸收？

这套追问能帮你判断：一个"新潮技术"是**真的解决了结构性问题**，还是**只是又一次徒劳的重复造轮子**。

### 7.2 它是一个判断"系统抽象该不该升级"的视角

反过来，当 OS 或语言标准库设计者看到应用层正在大规模"自己做"，那通常意味着：**系统抽象已经跟不上时代了**。

这时候不是去指责应用开发者"不尊重系统"，而是要思考：**是不是该把这套范式吸收进系统本身了？**

WinUI、Compose、SwiftUI 是这个思考的产物。msquic 内置、内核 QUIC、QUIC offload 也是。

### 7.3 它揭示了软件架构的一条底层规律

**软件世界里没有永久的"正确分层"，只有临时的"合理分层"。** 需求变了、规模变了、硬件变了、生态变了，原本合理的分层就会变得不合理，于是反叛开始，库化发生，新默认产生，然后新的合理分层又开始酝酿下一次反叛。

这是一条比"UI 框架之争"、"协议之争"更大的规律。它关于**系统与应用之间权力的动态再分配**。

---

## 结语：每一次反叛，都是下一代的体制

把文章开头那句话再写一遍，这次把两边都写上：

> **Flutter 对 UI 做的事，就是 QUIC 对 TCP 做的事——它们都在把"原本由系统提供的一整个子系统"以库的形式拉进应用，换取一致性、迭代速度、精细控制与性能极限，代价是生态割裂、体积膨胀与调试困难。**

然后再把后半段补上：

> **而这些反叛，只要成功，最终都会被系统吸收，成为下一代的默认——就像 TCP 当年一样。**

自绘 UI 与用户态协议栈，表面上一个是图形一个是网络，中间隔着整个操作系统的抽象层级；但骨子里，它们是同一个故事的两个章节。看懂一个，就几乎看懂了另一个。

更重要的是，看懂这两个，你就能在别的地方认出同一个故事：FUSE、Go runtime 的协程调度、eBPF、Service Mesh 的数据面、浏览器里的 WebGPU、AI 框架里的自制算子调度器……

它们都在重复同一句话：

**"系统给的那套抽象不够用了，我自己来。"**

——然后，在成功的那天，变成下一代的"系统抽象"。
